1. What is prerender? What is Prerender V2?

Prerender (simple)

👉 Prerender = generate full HTML of the page in advance for Google

Instead of Google running your JS:

  • Server runs everything
  • Sends ready-made HTML

So bots see full content instantly.

From official explanation:

  • It creates a fully rendered HTML snapshot for crawlers
  • Bots receive ready content without executing JS

Why it exists

Because:

  • JS sites load content late
  • Bots may miss content

👉 Prerender fixes that.

Prerender V2 (in your debug)

  • It’s just a newer version of the prerender service
  • Same idea, better engine (performance, caching, rendering)

👉 Meaning:

  • Your page was rendered by a server-side rendering service for bots

3. Why official bundles are slow?

Good question — and your assumption is slightly off.

👉 “Official” ≠ “fast”

Reasons:

a. Data/API complexity (official explanation aligned)

  • PDP pulls data through Item + Product modules using Search API and backend data ([docs.oracle.com](https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_4786239896.html?utm_source=chatgpt.com))
  • These are real-time data fetches (item, options, etc.)

👉 Even simple-looking features still depend on backend calls

b. Too many real-time API calls is a known performance issue

  • High-volume real-time API calls instead of optimized/cached calls slow NetSuite ([randgroup.com](https://www.randgroup.com/insights/oracle-netsuite/erp/ultimate-netsuite-optimization-and-performance-guide/?utm_source=chatgpt.com))

👉 Even if you have:

  • 0 reviews
  • 4 related items

The system still:

  • Executes service
  • Queries backend
  • Processes response

c. Official best practice: reduce unnecessary runtime work

  • Performance depends on both frontend + backend combined ([tavanoteam.com](https://tavanoteam.com/suitecommerce-advanced/suitecommerce-performance-optimization-guide/?utm_source=chatgpt.com))
  • Heavy logic, scripts, or data fetching impacts load time

👉 Meaning:

Even official bundles should be reviewed if not needed

d. Best practice: move non-critical logic away from real-time

  • Non-critical logic should be shifted away from real-time execution ([randgroup.com](https://www.randgroup.com/insights/oracle-netsuite/erp/ultimate-netsuite-optimization-and-performance-guide/?utm_source=chatgpt.com))

👉 For your case:

  • Reviews (empty) → still real-time call ❌
  • Related items (few) → still service call ❌

e. General SuiteCommerce guidance

  • PDP should return information “quickly and efficiently” ([docs.oracle.com](https://docs.oracle.com/en/cloud/saas/netsuite/ns-online-help/section_4786239896.html?utm_source=chatgpt.com))

👉 If extra modules slow it:

→ they go against this principle

Final simple takeaway:

👉 Official bundles are not slow because they are bad

👉 They are slow because:

  • They always execute (even if data is small)
  • They depend on backend APIs
  • They are not SEO/prerender optimized by default
  • Real-time calls are a known performance bottleneck

👉 So best practice is:

Audit each bundle → keep only what is truly needed during initial load

ChatGPT

7. How to skip or delay things in prerender (official guidance)

From how prerender works:

👉 Server detects bot → sends prerendered HTML instead of full app

Ways (based on official prerender concepts)

1. Control what is rendered for bots

  • Detect crawler (user-agent)
  • Serve simplified version

👉 Only include:

  • Main content
  • SEO-important elements

2. Avoid unnecessary JS execution

Since prerender:

  • Executes JS in headless browser

👉 Reduce:

  • Async features
  • Non-critical scripts

3. Use caching

  • First render is slow (cache miss)
  • Later → fast (cached HTML)

4. Reduce dynamic content for bots

Official SEO guidance:

  • Bots need content, not interactivity
  • Serve static HTML snapshot

Practical interpretation for your case

To skip during prerender:

  • Don’t trigger:
    • chatbot
    • reviews
    • related items
  • Or lazy-load after initial render
